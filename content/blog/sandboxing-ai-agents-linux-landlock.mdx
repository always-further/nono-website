Linux 5.13 introduced Landlock, an unprivileged Linux Security Module that allows any process to restrict its own filesystem access — without root privileges, without containers, and without any special configuration.

This is exactly the kind of primitive that makes AI agent sandboxing practical. When an AI coding agent runs on your machine, it has access to everything you do: SSH keys, cloud credentials, source code across every project. Landlock lets us change that.

## Why Landlock matters for AI agents

Traditional sandboxing approaches like Docker or VMs introduce significant overhead. They require a daemon, image management, networking configuration, and volume mounts just to let an agent edit a file. For interactive coding agents that need sub-second response times, this overhead is unacceptable.

Landlock operates at the kernel level with zero overhead after the initial policy application. There are no context switches to a hypervisor, no filesystem overlay merges, no network namespace juggling.

<Warning>
  Landlock requires Linux kernel 5.13 or newer. Run `uname -r` to check your kernel version before proceeding.
</Warning>

## How Landlock works

Landlock operates through a ruleset that defines allowed filesystem operations. Once applied, the ruleset is **irrevocable** for the lifetime of the process and all of its children.

```rust filename="src/sandbox/landlock.rs"
use landlock::{
    ABI, Access, AccessFs, PathBeneath,
    PathFd, Ruleset, RulesetAttr,
};

pub fn apply_sandbox(allowed: &[PathBuf]) -> Result<()> {
    let abi = ABI::V5;
    let access = AccessFs::from_all(abi);

    let mut ruleset = Ruleset::default()
        .handle_access(access)?
        .create()?;

    for path in allowed {
        let fd = PathFd::new(path)?;
        ruleset = ruleset.add_rule(
            PathBeneath::new(fd, access)
        )?;
    }

    ruleset.restrict_self()?;
    Ok(())
}
```

The key properties that make Landlock ideal for agent sandboxing:

- **Unprivileged**: No root, no capabilities, no suid binaries
- **Irrevocable**: Once applied, the sandbox cannot be loosened — only tightened
- **Inherited**: Child processes inherit the sandbox automatically
- **Zero overhead**: Kernel-level enforcement with no runtime cost

<Tip>
  nono applies Landlock before the agent process starts, ensuring the sandbox is active before any agent code executes. This eliminates race conditions between sandbox setup and agent initialization.
</Tip>

## Comparison with other approaches

| Approach | Root required | Overhead | Escape difficulty | Filesystem granularity |
|----------|--------------|----------|-------------------|----------------------|
| Landlock | No | None | Kernel-level | Per-path |
| seccomp | No | Minimal | Kernel-level | None (syscall only) |
| Docker | Yes (daemon) | High | Container escape | Volume mounts |
| Firejail | No (suid) | Low | Profile bypass | Profile-based |

nono uses both Landlock and seccomp simultaneously — Landlock for filesystem restrictions and seccomp for syscall filtering. This provides defense in depth at the kernel level.

## The nono approach

When you run an agent with nono, the sandbox policy is defined declaratively:

```bash
nono run --allow ~/projects/myapp -- claude
```

This single command:

1. Parses the policy into Landlock filesystem rules
2. Applies the Landlock ruleset to the current process
3. Layers seccomp filters on top for syscall restriction
4. Execs the agent process, which inherits both sandboxes

The agent can read and write files under `~/projects/myapp` but cannot access anything else on the filesystem. Attempts to read `~/.ssh/id_rsa` or `~/.aws/credentials` will fail with `EACCES`.

<Note>
  On macOS, nono uses the Seatbelt sandbox framework instead of Landlock. The policy syntax remains identical — nono translates the declarative policy to the appropriate kernel mechanism for each platform.
</Note>

## What comes next

Filesystem isolation is the foundation, but AI agents also make network requests, spawn subprocesses, and interact with system services. Each of these vectors requires its own enforcement mechanism layered on top of the filesystem sandbox.

The goal is straightforward: give AI agents exactly the access they need and nothing more, enforced at the kernel level where it cannot be bypassed.
